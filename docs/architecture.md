# Technical Architecture

## Overall Structure

Rayport consists of two parts:

- A precompiled WebAssembly runtime containing the CPython interpreter, raylib
  engine, and CFFI bindings. This artifact is built independently of user game
  code.
- A Python CLI that collects game files, generates the HTML page, and starts
  the development server.

Running `rayport build ./my-game/` produces a `build/` directory containing:

- `index.html`: the page that hosts the game.
- `main.wasm` (14.1 MiB): the CPython and raylib WebAssembly binary.
- `main.js` (286.4 KiB): Emscripten glue code that loads WebAssembly,
  initializes memory, and configures the canvas.
- `main.data` (3.0 MiB): the Python standard library (`.pyc`) loaded into the
  virtual filesystem at startup.
- `game.tar.gz`: the user's game code and assets.

Upload `build/` to any static web server to run the game.

## Execution Flow

After the browser loads `index.html`:

1. The browser downloads `main.js`, `main.wasm`, and `main.data`.
2. Emscripten initializes WebAssembly and loads the Python standard library
   from `main.data` into the virtual filesystem.
3. JavaScript in `index.html` downloads `game.tar.gz`, extracts it, and writes
   it to `/usr/local/game/` in the virtual filesystem.
4. The CPython interpreter starts and executes
   `/usr/local/lib/python3.12/launcher.py`.
5. `launcher.py` adds `/usr/local/game/` to `sys.path` and executes `main.py`.
6. The game imports the built-in raylib bindings with
   `from raylib import *`.
7. The game loop starts. Each call to `WindowShouldClose()` lets ASYNCIFY pause
   WebAssembly so the browser can update the display.

## Static Linking Architecture

All C code is statically linked into one WebAssembly binary:

- CPython 3.12.11 (`libpython3.12.a`).
- raylib 5.5 (`libraylib.a`).
- CFFI bindings (`_raylib_cffi.o`), generated by `gen_cffi.py` and compiled
  with Emscripten.
- The CFFI backend (`_cffi_backend.o`), compiled from the cffi package source.
- libffi type stubs (`ffi_type_stubs.o`), which provide type descriptors and
  stub functions.
- Web stubs (`web_stubs.o`) for raylib functions unavailable on the Web
  platform.

Rayport modifies CPython's `Modules/config.c` to register `_raylib_cffi` and
`_cffi_backend` in `_PyImport_Inittab`. Python therefore obtains them as
built-in modules without accessing the filesystem.

The build does not use Emscripten's dynamic `-sMAIN_MODULE` mode. This avoids
PIC relocation problems caused by raylib's global RLGL state. The tradeoff is
that Python cannot load `.so` extension modules at runtime, so required native
modules must be compiled into the runtime.

## ASYNCIFY Game Loop

Browser JavaScript is single-threaded. A continuously running Python
`while not WindowShouldClose()` loop would freeze the browser.

raylib calls `emscripten_sleep(12)` inside `WindowShouldClose()`, pausing for
12 milliseconds and returning control to the browser. Emscripten's ASYNCIFY
feature saves the complete C call stack, including CPython's `ceval.c`
evaluation loop, and restores it after the pause.

The linker flags `-sASYNCIFY -sASYNCIFY_STACK_SIZE=65536` enable this behavior.
User game code requires no changes.

## Virtual Filesystem

CPython inside WebAssembly uses Emscripten's in-memory MEMFS. Files come from
two sources:

- `main.data` (preload): the Python standard library and raylib Python package
  (`__init__.py`, `colors.py`, `defines.py`, `enums.py`, and `version.py`),
  loaded into `/usr/local/lib/python3.12/` during initialization.
- `game.tar.gz` (runtime fetch): user game code and assets, downloaded by
  JavaScript in `index.html` and extracted into `/usr/local/game/`.

## raylib Python Package Imports

The original raylib-python-cffi `__init__.py` uses the relative import
`from ._raylib_cffi import ffi, lib`, expecting `_raylib_cffi` to be a child of
the `raylib` package. In the WebAssembly runtime, `_raylib_cffi` is a top-level
built-in module.

The WebAssembly-specific `__init__.py` instead:

1. Runs `import _raylib_cffi`.
2. Assigns `sys.modules['raylib._raylib_cffi'] = _raylib_cffi`, making the
   import system recognize it as a child of `raylib`.
3. Copies every public symbol from `_raylib_cffi.lib` into `globals()`, making
   C function names available through `from raylib import *`.

## Build Environment

- macOS Darwin on ARM64.
- Emscripten 3.1.61. Version 6.0.2 uses resizable `ArrayBuffer` objects that
  are incompatible with the browser Crypto API used by this runtime.
- CPython 3.12.11.
- raylib 5.5.
- Python 3.13 and 3.14 as verified host CLI versions.

The `Makefile` automates the build. Local overrides belong in the ignored
`build.conf`; see `build.conf.example`.

## Directory Structure

- `src/rayport/`: Python CLI source.
- `rayport.toml`: optional Web presentation and package rules for each game.
- `src/rayport/runtime/`: precompiled `main.wasm`, `main.js`, and `main.data`
  files published as Python package data.
- `stubs/`: build-time C stubs and helper scripts, including `gen_cffi.py`,
  `web_stubs.c`, `ffi_type_stubs.c`, and `launcher.py`.
- `.cache/`: ignored intermediate build artifacts.

Source checkouts and wheel installations both locate the three runtime
artifacts through `importlib.resources`. The `RAYPORT_RUNTIME` environment
variable can point to a custom runtime.

## Known Limitations

The Python interpreter inside WebAssembly does not support:

- Network connections through `socket`, `urllib`, or `requests`.
- Child processes through `fork`, `exec`, or `subprocess`.
- Native threads because of `SharedArrayBuffer` limitations.
- Physical filesystem access; only in-memory MEMFS is available.
- `ctypes`, `ssl`, or dynamically loaded `.so` modules.

These limitations do not affect raylib graphics, input, or audio, which run
through the C engine compiled into WebAssembly.

The runtime directly provides the C-style `raylib` module. Python-style names
such as `init_window` and `begin_drawing` come from `pyray`. During a game
build, Rayport analyzes imports reachable from `main.py` and copies `pyray`
and other required pure Python packages from the game's `.venv` or `venv`.
Native `.so`, `.pyd`, `.dll`, and `.dylib` files are not packaged.
